Event Loop, Microtasks, and I/O Phases

Question

Explain the Node.js event loop phases, microtasks, and how setImmediate, setTimeout(0), and Promises are ordered.

Answer

Node.js runs an event loop with phases: timers, I/O callbacks, poll, check, and close callbacks. Microtasks (Promise.then, queueMicrotask) run after the current call stack and between phases, so they usually execute before timers and setImmediate.

Illustration


Call Stack
  ↓
Microtasks (Promises, queueMicrotask)
  ↓
Event Loop Phases
  - Timers (setTimeout, setInterval)
  - I/O callbacks
  - Poll
  - Check (setImmediate)
  - Close callbacks

Code Example


setTimeout(() => console.log('timeout'), 0);
setImmediate(() => console.log('immediate'));
Promise.resolve().then(() => console.log('promise'));
console.log('sync');

// Typical order
// sync
// promise
// timeout or immediate (order depends on I/O timing)

Senior Notes

  • Microtasks always run before the next phase.
  • Inside an I/O callback, setImmediate often fires before setTimeout(0).
  • Event loop lag can be measured with setInterval or perf_hooks.

Streams and Backpressure

Question

How do you handle large uploads/downloads without high memory usage?

Answer

Use streams and backpressure. Avoid buffering large payloads in memory. Pipe a readable stream into a writable stream so Node can apply flow control.

Illustration


Client → Readable Stream → Transform → Writable Stream
            ↑ backpressure controls flow

Code Example


const fs = require('fs');
const http = require('http');

http.createServer((req, res) => {
  const read = fs.createReadStream('./large-file.bin');
  read.pipe(res);
}).listen(3000);

Senior Notes

  • Use stream.pipeline to handle errors reliably.
  • Tune highWaterMark for throughput vs memory.
  • For uploads, pipe request stream into storage or hashing directly.

Scaling Node.js: Cluster, Worker Threads, and Horizontal Scale

Question

Node.js is single‑threaded. How do you scale CPU‑bound work and handle traffic growth?

Answer

Use worker threads for CPU‑bound tasks and multiple Node processes for parallel request handling. Horizontally scale with containers and a load balancer.

Illustration


Load Balancer
  → Node Process 1
  → Node Process 2
  → Node Process 3

CPU Task → Worker Thread Pool

Code Example (Worker Thread)


const { Worker } = require('worker_threads');

function runJob(data) {
  return new Promise((resolve, reject) => {
    const worker = new Worker('./worker.js', { workerData: data });
    worker.on('message', resolve);
    worker.on('error', reject);
  });
}

Senior Notes

  • Prefer I/O in the main process, CPU work in workers.
  • Use cluster or process managers for multi‑core utilization.
  • Monitor event loop lag to detect CPU saturation.

Database Transactions and Isolation

Question

How do you ensure consistency with concurrent requests in a Node.js API?

Answer

Use ACID transactions at the database level and pick a suitable isolation level. Avoid app‑level locks for correctness.

Illustration


T1: Read balance=100 → Withdraw 80 → Commit
T2: Read balance=100 → Withdraw 80 → Commit

If not isolated → balance becomes -60 (lost update)

Code Example (PostgreSQL with node‑pg)


const { Pool } = require('pg');
const pool = new Pool();

async function transfer(fromId, toId, amount) {
  const client = await pool.connect();
  try {
    await client.query('BEGIN');
    await client.query('UPDATE accounts SET balance = balance - $1 WHERE id=$2', [amount, fromId]);
    await client.query('UPDATE accounts SET balance = balance + $1 WHERE id=$2', [amount, toId]);
    await client.query('COMMIT');
  } catch (e) {
    await client.query('ROLLBACK');
    throw e;
  } finally {
    client.release();
  }
}

Senior Notes

  • Use SERIALIZABLE or explicit row locks for critical invariants.
  • Avoid deadlocks by ordering updates consistently.
  • Include retries for serialization failures.

Idempotent APIs

Question

How do you make a POST request safe against retries and network failures?

Answer

Use idempotency keys. Store the result for a request token so repeated calls return the same outcome without double‑writing.

Illustration


Client → POST /charge (Idempotency-Key: abc123)
Server → if abc123 exists → return stored result

Code Example (Conceptual)


const cache = new Map();

async function createCharge(req, res) {
  const key = req.headers['idempotency-key'];
  if (key && cache.has(key)) return res.json(cache.get(key));

  const result = { status: 'charged', id: Date.now() };
  if (key) cache.set(key, result);
  res.json(result);
}

Senior Notes

  • Use Redis for distributed idempotency with TTLs.
  • Record request fingerprints to avoid replay with altered payloads.
  • Make downstream side effects idempotent too.

Caching Strategies (Read‑Through, Write‑Through)

Question

How do you reduce database load for hot endpoints?

Answer

Use a cache layer with TTLs and invalidation. Read‑through caches fetch on miss and store. Write‑through caches update the cache on writes.

Illustration


Request → Cache Hit → Return
       → Cache Miss → DB → Cache Store → Return

Code Example (Redis Read‑Through)


async function getUser(id, redis, db) {
  const key = `user:${id}`;
  const cached = await redis.get(key);
  if (cached) return JSON.parse(cached);

  const user = await db.getUser(id);
  await redis.set(key, JSON.stringify(user), 'EX', 60);
  return user;
}

Senior Notes

  • Prevent cache stampedes with locks or request coalescing.
  • Use short TTLs for volatile data.
  • Cache negative results carefully to avoid stale errors.

Rate Limiting and Abuse Prevention

Question

How do you prevent brute‑force and abuse at the API layer?

Answer

Implement rate limiting with a token bucket or sliding window. Enforce per IP, per user, and per route limits.

Illustration


Token Bucket
- refill rate: 5 req/s
- burst: 20

Code Example (In‑Memory Token Bucket)


const buckets = new Map();

function allow(ip) {
  const now = Date.now();
  const bucket = buckets.get(ip) || { tokens: 20, last: now };
  const refill = ((now - bucket.last) / 1000) * 5;
  bucket.tokens = Math.min(20, bucket.tokens + refill);
  bucket.last = now;

  if (bucket.tokens >= 1) {
    bucket.tokens -= 1;
    buckets.set(ip, bucket);
    return true;
  }
  buckets.set(ip, bucket);
  return false;
}

Senior Notes

  • Use Redis or a shared store in distributed setups.
  • Add separate burst and sustained limits.
  • Combine with WAF rules and bot detection.

Message Queues and At‑Least‑Once Delivery

Question

How do you handle background jobs without blocking requests?

Answer

Push work to a queue and process asynchronously. Expect at‑least‑once delivery and make handlers idempotent.

Illustration


API → Queue → Worker(s) → DB/Email
           ↑ retries, at‑least‑once

Code Example (Idempotent Worker)


async function handler(job) {
  if (await isProcessed(job.id)) return;
  await doWork(job.data);
  await markProcessed(job.id);
}

Senior Notes

  • Avoid double execution with dedup or idempotent writes.
  • For Kafka, manage offsets and rebalances carefully.
  • Use dead‑letter queues for poison messages.

Security: Input Validation and Injection Prevention

Question

How do you secure Node.js APIs from injection and malformed input?

Answer

Validate inputs with schemas, use parameterized queries, and never build SQL with string concatenation.

Illustration


Request → Validate → Sanitize → Execute

Code Example (Zod Validation)


const { z } = require('zod');

const schema = z.object({
  email: z.string().email(),
  age: z.number().int().min(13)
});

function validate(body) {
  return schema.parse(body);
}

Senior Notes

  • Use allow‑lists for fields.
  • Add output encoding and CSP headers.
  • Avoid logging sensitive data (tokens, passwords, PII).

Observability: Logs, Metrics, and Traces

Question

How do you debug latency spikes in production?

Answer

Use structured logs, request IDs, metrics (P95, P99), and distributed tracing. Correlate logs with traces to isolate slow dependencies.

Illustration


Request ID → Log → Trace → Metrics

Code Example (Request ID Middleware)


const { randomUUID } = require('crypto');

function requestId(req, res, next) {
  req.id = randomUUID();
  res.setHeader('x-request-id', req.id);
  next();
}

Senior Notes

  • Track event loop lag and memory usage.
  • Alert on error rates and saturation.
  • Tag metrics with route and dependency labels.

CAP Theorem (Consistency, Availability, Partition Tolerance)

Question

What does CAP mean, and how does it affect backend system design?

Answer

CAP says in the presence of a network partition, a distributed system must choose between **Consistency** and **Availability**. Partition tolerance is required in real networks. You decide whether to favor consistency (CP) or availability (AP) during partitions.

Illustration


Network Partition
  ↙         ↘
Node A    Node B

Choose: Consistency or Availability during partition

Practical Examples

  • CP: Financial transactions, inventory reservation.
  • AP: Analytics dashboards, social feeds, caching layers.

Node.js Code Example (AP‑style cache fallback)


async function getProfile(id, cache, db) {
  const cached = await cache.get(id);
  if (cached) return JSON.parse(cached);

  try {
    const user = await db.getUser(id);
    await cache.set(id, JSON.stringify(user), 'EX', 60);
    return user;
  } catch (e) {
    // AP choice: return stale or empty during DB partition
    return cached ? JSON.parse(cached) : { id, status: 'stale' };
  }
}

Senior Notes

  • CP systems may reject writes during partitions.
  • AP systems may return stale data and reconcile later.
  • You often mix CP and AP at different layers.

Database Indexing

Question

How do database indexes work, and how do you decide what to index?

Answer

Indexes are data structures (typically B‑trees or hashes) that speed up lookups at the cost of extra storage and slower writes. You index columns used in filters, joins, and sort operations, especially for high‑selectivity queries.

Illustration


Table Scan:  O(n)
Index Seek: O(log n)

Example Query


SELECT * FROM orders
WHERE user_id = 42 AND created_at > NOW() - INTERVAL '7 days'
ORDER BY created_at DESC

**Good index:** (user_id, created_at DESC)

Node.js Code Example (Query Optimization)


async function getRecentOrders(db, userId) {
  const sql = `
    SELECT id, total, created_at
    FROM orders
    WHERE user_id = $1
    ORDER BY created_at DESC
    LIMIT 20
  `;
  return db.query(sql, [userId]);
}

Senior Notes

  • Composite indexes should match filter order.
  • Avoid over‑indexing write‑heavy tables.
  • Use EXPLAIN to verify plan and index usage.

System Design Deep Dives (Senior Level)

Below are two common FAANG‑level system design prompts with Node.js‑oriented considerations and code snippets.

---

A. Design a URL Shortener

Requirements

  • Create short URLs and redirect quickly.
  • Handle high read traffic.
  • Low latency and high availability.

Core Design

  • API: POST /shorten, GET /:code
  • Storage: key‑value store for code → long_url.
  • Generate code with base62 and collision checks.
  • Cache hot keys in Redis.

Illustration


Client → API → Cache → DB
        ← Redirect

Node.js Code Sketch


function base62Encode(num) {
  const chars = '0123456789abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ';
  let out = '';
  while (num > 0) {
    out = chars[num % 62] + out;
    num = Math.floor(num / 62);
  }
  return out || '0';
}

async function createShortUrl(db, longUrl) {
  const id = await db.nextId();
  const code = base62Encode(id);
  await db.put(code, longUrl);
  return code;
}

Senior Notes

  • Use append‑only ID generation to avoid hot partitions.
  • Add expiration for one‑time or temporary links.
  • Store redirects in CDN or edge cache for ultra‑low latency.

---

B. Design a Rate‑Limited Notification Service

Requirements

  • Accept notification requests from multiple services.
  • Deliver via email, SMS, push.
  • Respect per‑user rate limits and retries.

Core Design

  • API: POST /notify
  • Queue for async processing.
  • Worker pool with per‑user throttling.
  • Deduplication using idempotency keys.

Illustration


API → Queue → Worker → Provider
              ↑ rate limits

Node.js Code Sketch


async function processJob(job, limiter, provider) {
  const ok = await limiter.consume(job.userId, 1);
  if (!ok) throw new Error('rate limit');

  await provider.send(job.payload);
}

Senior Notes

  • Use exponential backoff and DLQs for failures.
  • Track delivery status with a state machine.
  • Separate ingestion from delivery for resilience.
